# Premium cinematic storytelling website with Three.js shader background

A static agency landing page built as a dark-themed, scroll-driven cinematic experience. Four narrative "chapter" sections animate into view via GSAP ScrollTrigger, layered over a fullscreen Three.js simplex-noise shader that responds to cursor position. Lenis handles smooth scrolling with GSAP ticker integration. The services section uses glassmorphism cards on a responsive grid. All libraries (Three.js r160, GSAP 3.12.5, Lenis 1.0.42) load from CDN with no build step.

**Watch for:** The Three.js render loop runs unconditionally at full framerate even when the tab is backgrounded or the canvas is occluded—no visibility-based throttling exists (confirmed). The animation system initializes without checking whether CDN scripts actually loaded, so a blocked CDN leaves the page as a blank unscrolled state with all content invisible (confirmed). Mobile navigation is hidden with `display: none` and no hamburger or alternative is provided, leaving the nav menu completely inaccessible on small viewports (confirmed).

## High-level view

The page is structured as a single HTML document with four JS modules loaded sequentially: smooth-scroll, three-background, animations, and a main orchestrator. The orchestrator assumes all three dependencies exist globally and calls their init functions synchronously. There's no error boundary; a CDN failure cascades into a fully broken page because all content starts at `opacity: 0` and only GSAP reveals it.

The Three.js render loop never pauses—no visibility check, no intersection observer. On mobile this means continuous GPU work for an effect that's largely occluded by chapter gradient overlays and a grain texture. The resize handler also misses re-applying `setPixelRatio`, so device orientation changes render at stale density.

The responsive strategy relies on CSS media queries at 1024px, 768px, and 480px breakpoints. The desktop nav menu disappears entirely at 768px with no mobile replacement, making the site unnavigable on phones.

<details>
<summary>Issues (5)</summary>

1. **Uncapped render loop** — The Three.js `requestAnimationFrame` loop renders every frame regardless of page visibility. Add a `visibilitychange` listener to pause rendering when the tab is hidden, and consider an IntersectionObserver to stop when the canvas is scrolled off-screen.
2. **No CDN load fallback** — All content starts at `opacity: 0` and relies on GSAP to reveal. If any CDN script fails to load, the page is invisible. Guard init calls with `typeof` checks and add a CSS fallback that removes `opacity: 0` when a `no-js` or `load-failed` class is present.
3. **Mobile nav inaccessible** — `nav-menu` is `display: none` below 768px with no hamburger toggle. Users on phones/tablets cannot navigate between sections. Add a mobile menu interaction or at minimum make anchor links accessible another way.
4. **No resource cleanup** — The Three.js renderer, geometry, material, and event listeners are allocated once and never disposed. For a single-page static site this is acceptable, but if this page is ever embedded in a SPA or navigation context, it will leak GPU resources and listeners.
5. **Stale pixel ratio on orientation change** — `Math.min(window.devicePixelRatio, 1.5)` is set once at init. The resize handler doesn't re-call `setPixelRatio`, so a device orientation change on a tablet renders at the wrong density until a full reload.

## Three.js render lifecycle

The render loop runs unconditionally:

```javascript
function animate() {
  requestAnimationFrame(animate);
  // ... update uniforms ...
  renderer.render(scene, camera);
}
animate();
```

There is no `document.visibilitychange` handler to pause when the tab is inactive, and no intersection check to stop when the user scrolls past the hero into content-heavy sections where the background is heavily occluded by chapter gradient overlays. On a mobile device with a 60Hz display, this means continuous GPU work for an effect that's barely visible beneath `opacity: 0.03` grain and `rgba` chapter backgrounds.

The resize handler updates renderer size and resolution uniform but does not re-apply `setPixelRatio`. If `devicePixelRatio` changes (device orientation on some tablets, or moving a window between displays on desktop), the renderer will continue at the stale ratio.

## Content visibility dependency on JavaScript

Every animated element starts hidden via CSS:

```css
.reveal-text, .reveal-up, .fade-in {
  opacity: 0;
  will-change: transform, opacity;
}
```

The CSS includes a comment suggesting `body.loaded` as a JS-fail fallback, but the actual CSS rule under `body.loaded` is empty—it does nothing. If GSAP fails to load (CDN blocked by adblocker, corporate proxy, network error), the user sees a completely blank page.

A minimal fix would be a `<noscript>` style block or a timeout-based class that forces `opacity: 1` on all animated elements if GSAP hasn't initialized within a reasonable window.

## Mobile navigation gap

The nav menu is hidden entirely at 768px:

```css
@media (max-width: 768px) {
  .nav-menu {
    display: none;
  }
}
```

No hamburger icon, no drawer, no alternative navigation mechanism exists. The five anchor links (Vision, Intelligence, Innovation, Services, Contact) become completely unreachable on mobile. Users can still scroll through content linearly, but the "Get in Touch" CTA link and section-jumping capability are lost.

## Hero animation race on slow connections

The hero animation timeline fires on page load with a 0.3s delay, but if Lenis hasn't finished initializing its scroll position by then (or if the user has already scrolled during load), the hero elements might animate in while already partially scrolled past, creating a visual flash. This is unlikely on a fresh load but could manifest on slow connections where the user starts scrolling before all scripts complete.

## Grain overlay z-index stacking

The grain overlay sits at `z-index: 1000`, above the nav (`z-index: 900`). Any future interactive overlays (modals, dropdowns, mobile menus) would need `z-index > 1000` to appear above the grain. If a mobile nav drawer is added to fix the navigation gap, this stacking context needs to be accounted for.
